# Continuous Improvement

## Objetivo
Dar criterios para reconocer gaps reales del sistema y describirlos con evidencia suficiente.

## Señales fuertes
- La misma pregunta al humano aparece de forma recurrente.
- Una skill deja huecos que obligan a improvisar una y otra vez.
- La terminologia cambia entre HU, FR, DD, TD o UD.
- El flujo rebota repetidamente entre dos fases por una ambiguedad sistemica.
- Faltan reglas transversales para un caso comun.
- Una mejora concreta evitaria bloqueos o gates fallidos repetidos.

## Evidencia minima
- Contexto donde aparece el gap.
- Frecuencia o repeticion observable.
- Impacto en tiempo, calidad o numero de vueltas.
- Propuesta concreta y acotada.
- Riesgo o tradeoff de aplicar esa mejora.

## No usar para
- Casos aislados.
- Preferencias personales sin impacto repetible.
- Cambios de implementación de una feature concreta.
- Decisiones que pertenecen al proyecto actual y no al sistema de agentes.

## Resultado esperado
- Una propuesta pequeña, verificable y fácil de revisar.
- Un cambio pensado para reducir ambiguedad futura, no para reabrir la fase actual.
